# Karpathy 的 LLM Wiki：让 LLM 帮你维护一套个人 Wiki

> 原文：https://cloud.tencent.com.cn/developer/article/2653360
> 发布于 2026-06-15

## 核心问题

很多人用 LLM 处理资料的常见路径：
- 上传一堆 PDF、网页、笔记
- 问一个问题
- 让模型临时检索相关片段
- 再在当前会话里拼答案

问题：
- 每次都要从头翻资料
- 模型每次都要重新拼接上下文
- 上次问过的问题不会沉淀下来
- 文档之间的关系也不会自己长出来

**LLM Wiki 补的，就是中间这层。**

## 三层架构

1. **raw sources** — 原始资料（文章、论文、图片、数据文件），保持不变
2. **wiki** — 一整套由 LLM 维护的 markdown 页面（总结、交叉引用、主题页、人物页、概念页）
3. **schema** — 告诉 LLM 这套 wiki 该怎么维护的规则文件（CLAUDE.md 或 AGENTS.md）

原始资料是 source of truth，wiki 是中间产物。平时读的是 wiki，但 wiki 本身要持续回看、持续修、持续吸收新资料。

## 与传统 RAG 的差别

- RAG：每次问问题都重新找片段
- LLM Wiki：先把知识整理成稳定页面，再在这些页面上继续问

**把"当场拼答案"改成"先把知识库存起来，再在库上工作"。**

## 体验变化

因为 wiki 已经先整理过一轮：
- 交叉引用已经建好
- 冲突信息已经标出来
- 主题总结已经积累下来
- 之前问过的问题可以继续沉淀回 wiki

后面的工作不是反复"重新理解原始资料"，而是在已有结构上继续往下长。

**Karpathy 的比喻**：Obsidian 像 IDE，LLM 像程序员，wiki 像代码库。

## 适用场景

- 个人知识管理
- 长期研究某个主题
- 边读一本书边建角色、主题、线索页
- 团队内部 wiki
- 竞品分析、尽调、旅行规划、课程笔记

共同点：资料会不断累积，问题不会只问一次，人想要的不只是"找到片段"而是"把结构慢慢整理出来"。

## 三大操作

### 1. Ingest
把新资料放进 raw sources，让 LLM 处理：
- 读资料
- 跟你讨论重点
- 生成摘要页
- 更新 index
- 更新相关实体页和概念页
- 记进 log

Karpathy 偏好一次 ingest 一份资料，自己看着，让 LLM 决定该强调什么。

### 2. Query
问问题时先去 wiki 里找相关页，再综合回答。回答本身还可以回写成 markdown 页面、比较表、slide deck、chart、canvas。让"提问结果"也开始沉淀。

### 3. Lint
定期健康检查 wiki：
- 页面之间有没有冲突
- 有没有旧结论被新资料推翻
- 有没有孤立页
- 有没有该有但还没建的概念页
- 有没有缺交叉引用
- 哪些地方可能需要补新资料

## index.md 和 log.md

**index.md** — 内容导向的目录。页面链接、一行摘要、分类、可选元数据。模型回答问题时先读 index，再决定深入哪些页面。~100 个 source / 几百个页面的规模下可以跑顺。

**log.md** — 时间导向的记录。什么时候 ingest 了什么、什么时候做了 query、什么时候跑了 lint。前缀格式统一的话可以用 Unix 工具处理。

## 为什么成立

很多人放弃维护 wiki 是因为维护成本太高。LLM 的优势：
- 不会嫌烦
- 不会漏掉交叉引用
- 可以一次改十几个文件
- 维护成本接近于零

分工：人负责找资料、决定看什么、问对问题、判断材料意味着什么。LLM 负责整理、归档、链接、更新、维护一致性。

## 评论区补充

### 问题 1：知识整理会放大事实错误
对精度要求高的内容（API 参数、版本依赖），LLM 的小错误会被放大成知识污染。最稳的还是人工逐条核对。需要补事实核查层、可信度标记、引文约束、高风险页面人工 review。

### 问题 2：往训练数据方向走
不只用 markdown wiki 做检索和综合，继续整理成结构化训练数据，去本地微调模型。

## 一句话概括

**Karpathy 在试图把"知识检索"改成"知识编译"。**
